# cy-414 运动健身私教预约平台

请根据以下要求生成项目代码和 README。

## 项目主要需求

### 1. 项目概述

**运动健身私教预约平台（FitPro）** 是一款连接健身教练与学员的在线预约系统。学员可浏览教练资料、预约课程、记录体测数据；教练可管理课程时间、查看学员预约、制定训练计划。项目类型：**全栈 Web 应用**。

### 2. 核心实体（≥ 3 个，必须贯穿全栈）

- **User（用户）**
  - 字段：id, phone, password_hash, nickname, avatar, role（学员/教练/管理员）, gender, height, weight, created_at
  - 关键关系：教练可创建多个 Course，学员可创建多个 Booking 和 BodyMetric
  - 业务动作：注册/登录、修改资料、上传头像
  - **此实体必须贯穿全栈**

- **Course（课程）**
  - 字段：id, coach_id, title, description, duration, price, max_capacity, schedule（JSON 时间段）, status, created_at
  - 关键关系：隶属于教练（User.coach），被多个 Booking 引用
  - 业务动作：教练发布课程、学员查看课程、预约课程
  - **此实体必须贯穿全栈**

- **Booking（预约）**
  - 字段：id, user_id, course_id, schedule_time, status（待确认/已确认/已完成/已取消）, note, created_at
  - 关键关系：隶属于学员，引用 Course
  - 业务动作：预约课程、取消预约、教练确认、完成打卡
  - **此实体必须贯穿全栈**

- **BodyMetric（体测数据）**
  - 字段：id, user_id, weight, body_fat, muscle, bmi, record_date, created_at
  - 关键关系：隶属于学员
  - 业务动作：记录体测、查看趋势图、生成建议
  - **此实体必须贯穿全栈**

### 3. 核心页面（≥ 4 个，至少 2 个页面共用同一业务组件）

- **/dashboard（首页）**
  - 消费实体：Course、Booking、BodyMetric
  - 关键交互：推荐课程、即将开始的预约、体测趋势
  - 共享组件：`<CourseCard>`、`<BookingStatusBadge>`

- **/courses（课程广场）**
  - 消费实体：Course、User
  - 关键交互：课程搜索、教练筛选、课程详情、立即预约
  - 共享组件：`<CourseCard>`（与 /dashboard 共用）、`<CoachAvatar>`

- **/bookings（我的预约）**
  - 消费实体：Booking、Course
  - 关键交互：预约列表、取消预约、完成打卡
  - 共享组件：`<BookingStatusBadge>`（与 /dashboard 共用）、`<EmptyState>`

- **/body-metrics（体测记录）**
  - 消费实体：BodyMetric
  - 关键交互：新增体测、趋势图表、历史列表
  - 共享组件：`<MetricTrendChart>`、`<EmptyState>`（与 /bookings 共用）

- **/profile（个人中心）**
  - 消费实体：User
  - 关键交互：修改资料、查看会员信息
  - 共享组件：`<AvatarUploader>`

### 4. 横切关注点（≥ 2 个）

- **认证授权（JWT + RBAC）**
  - 触达文件层：
    - 数据库：`users` 表含 role 字段
    - 后端：`middlewares/auth.js`、`middlewares/roleCheck.js`、`utils/jwt.js`
    - 路由：教练专属接口、学员专属接口
    - 前端：`router/guards.ts`、`components/common/RoleGuard.tsx`、`stores/authStore.ts`

- **接口限流**
  - 触达文件层：
    - 后端：`middlewares/rateLimiter.js`
    - 路由：预约、登录等接口启用限流
    - 配置：`config/rateLimit.ts`

- **全局错误处理**
  - 触达文件层：
    - 后端：`middlewares/errorHandler.js`、`utils/AppError.js`
    - 前端：`utils/request.ts` 拦截器

### 5. 共享枚举/常量（≥ 2 组）

- **BookingStatus（预约状态）**
  - 值：PENDING = 'pending', CONFIRMED = 'confirmed', COMPLETED = 'completed', CANCELLED = 'cancelled'
  - 前端定义位置：`frontend/src/constants/booking.ts`
  - 后端定义位置：`backend/src/constants/booking.js`
  - 出现位置：Booking 模型、BookingService、BookingStatusBadge、预约列表、日志模板、错误码
  - **README 必须列出该枚举在前后端的所有出现位置**

- **UserRole（用户角色）**
  - 值：STUDENT = 'student', COACH = 'coach', ADMIN = 'admin'
  - 前端定义位置：`frontend/src/constants/user.ts`
  - 后端定义位置：`backend/src/constants/user.js`
  - 出现位置：User 模型、RBAC 中间件、路由权限、前端按钮显隐、路由守卫
  - **README 必须列出该枚举在前后端的所有出现位置**

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

1. **日志模块必须单独管理但全栈引用**
   - 后端必须有一个 `utils/logger.js`，所有 controller/service/middleware 都引用
   - 日志模板集中在 `constants/logTemplates.js`，包含至少 25 条模板
   - 任何业务字段变更（如 Booking 新增字段）都必须同步修改日志模板

2. **异常信息必须分散且层层透传**
   - 错误码集中在 `constants/errorCodes.js`，但 service 和 controller 都要手动包装
   - 错误 message 必须包含实体名、字段名、角色名，如 `"Booking[id=123] confirm failed: coach not match"`

3. **常量/工具类必须多处耦合**
   - `utils/formatters.ts` 同时包含日期格式化、价格格式化、状态文本、角色文本、BMI 等级文本
   - `constants/messages.ts` 同时包含前端提示文案、后端返回文案、日志文案
   - 修改一个文案可能同时影响前端显示、后端返回和日志内容

4. **预约状态机必须跨多处定义**
   - BookingStatus 的流转规则在 service 状态机、前端按钮显隐、日志模板、错误码、formatters 中同时存在
   - 新增一个状态值需要修改至少 10 处

5. **枚举必须前后端重复定义且被多处引用**
   - BookingStatus 和 UserRole 必须在前端 constants、后端 constants、类型定义、日志模板、错误码、formatters、Prisma schema 中同时存在
   - 新增一个枚举值时，必须修改至少 10 处文件

6. **牵一发动全身的验证标准**
   - 后续如果要求"给 Course 新增一个 'group' 课程类型"，必须触达：Prisma schema、Course 模型、前后端 constants、CourseService、CourseCard、CoachAvatar、formatters、日志模板、错误码、Dashboard 推荐逻辑、筛选器等 ≥ 10 个文件

### 12. 部署要求

这是前后端分离项目。

前端代码放在 `frontend/`：
- 页面、组件、路由、状态管理、API 请求、类型定义、样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
- 数据库脚本放在 `database/`。

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=fitpro`
- `frontend/Dockerfile`
- `backend/Dockerfile`
- `frontend/nginx.conf`
- `.env.example`

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: fitpro`
- 所有服务使用 `container_name` 命名容器，容器名带 `${COMPOSE_PROJECT_NAME:-fitpro}` 前缀
- 数据库数据使用命名卷持久化
- 前端通过 Nginx 反向代理后端 API（前端代码里调用 `/api`，由 Nginx 转发到 `backend:<port>`）
- 服务间通过 Docker 内部网络通信
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- 后端服务也配置 `healthcheck`
- 暴露端口：前端 `18414:80`，后端 `19414:3000`，数据库 `3306:3306`
- 所有可变配置通过环境变量注入
- 支持通过 `docker compose up -d` 一条命令启动

`.env` / `.env.example` 必须包含的键：
- `COMPOSE_PROJECT_NAME=fitpro`
- `DB_NAME=fitpro_db`、`DB_USER=fitpro_user`、`DB_PASSWORD=fitpro_pwd`、`DB_ROOT_PASSWORD=fitpro_root`
- `JWT_SECRET=change_me_to_a_long_random_string`
- `FRONTEND_PORT=18414`、`BACKEND_PORT=19414`、`DB_PORT=3306`

Nginx 反向代理要求（`frontend/nginx.conf`）：
- `location /api/` 反向代理到 `http://backend:3000/`
- 配置 `try_files $uri $uri/ /index.html;` 支持前端 SPA 路由
